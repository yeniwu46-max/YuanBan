# 鼋伴伴应用开发规划

## 1. 项目定位

“鼋伴伴”是一套面向社区居家养老场景的智慧养老应用与设备协同系统。项目总目标不是把老人作为被动监护对象，而是通过“无感监测，有感服务”实现安全守护、情感陪伴、自主健康管理和家庭/社区协同。

核心价值主张：

- 无感安全：通过毫米波雷达、环境传感器、SOS 按钮等设备降低老人主动佩戴和操作负担。
- 情感陪伴：以“小鼋”IP 承载语音反馈、情绪回应、节日祝福、健康鼓励等互动。
- 赋权增能：让老人能看懂自己的健康状态、主动记录生活和参与社区，而不是只被子女或机构监控。
- 家社协同：连接老人、子女、社区服务人员，形成日常关怀与异常应急闭环。

## 2. 目标用户与端侧角色

### 老人端

适配对象包括独居老人、空巢老人、低数字素养老人和轻度行动不便老人。设计优先级是大字、少步骤、语音优先、明确反馈、低认知负担。

核心需求：

- 快速确认“我现在是否安全”。
- 一键求助，触发子女和社区响应。
- 看懂健康数据和近期变化。
- 与子女联系、分享照片和心情。
- 被提醒、被鼓励，但不能有被监视感。

### 子女端

子女端以远程关怀和应急处理为主，不应做成复杂管理后台。

核心需求：

- 查看老人实时安全状态和异常记录。
- 接收跌倒、体征异常、SOS 等高优先级通知。
- 查看健康周报、心情趋势和互动建议。
- 远程设置提醒、协助老人完成配置。
- 发起语音/视频/图文关怀，参与“小鼋”成长。

### 社区/服务端

社区端面向社区工作人员、养老服务站、健康管理机构。

核心需求：

- 查看辖区老人异常告警队列。
- 分派上门服务、活动报名、便民预约。
- 记录响应结果，形成可追溯闭环。
- 统计设备在线率、服务完成率和风险人群。

## 3. 产品形态建议

项目要求移动端优先、网页端支持，同时申报书中强调微信小程序和轻量化入口。建议采用“三端一套核心能力”的方式。

首期推荐形态：

- 老人端/子女端：移动端 H5/PWA 或微信小程序优先。
- 网页端：响应式 Web，用于子女大屏查看和社区后台。
- 硬件端：先接入硬件模拟器，再接真实 ESP32-S3 + 毫米波雷达数据。

技术选型建议：

- 前端：uni-app + Vue 3 + TypeScript。原因是可同时覆盖 H5、微信小程序和后续 App 打包，适合大创团队快速迭代。
- UI：自建适老化组件规范，不直接套年轻化移动 UI。按钮热区不小于 44px，老人端正文字号建议 30px 级别，强对比、少层级。
- 后端：FastAPI 或 NestJS 均可。若团队偏 Python 和算法联调，优先 FastAPI；若团队偏前端全栈，优先 NestJS。
- 实时通信：MQTT 用于设备上报，WebSocket/SSE 用于前端实时状态刷新。
- 数据库：PostgreSQL 存业务数据，Redis 处理告警状态、消息去重和限流。
- 文件存储：对象存储保存相册、语音、报告附件。
- 消息通知：微信订阅消息、短信、电话 API 作为应急通知通道。

## 4. MVP 范围

第一版要先证明“能跑通养老守护闭环”，不要一开始同时做全量算法、硬件、社区生态和复杂积分商城。

### P0 必做

- 登录与角色绑定：老人、子女、社区三类角色；支持一个老人绑定多个子女。
- 老人首页：安全状态、SOS、大字健康摘要、小鼋情绪入口、家庭联络入口。
- 子女首页：老人在线状态、今日风险、最新体征、告警记录、远程提醒。
- 设备接入模拟器：模拟心率、呼吸、跌倒、异常静止、设备离线、SOS。
- 告警闭环：告警生成、推送、确认、处理、备注、关闭。
- 健康报告：日报/周报，包括心率、呼吸、异常次数、活动/互动摘要。
- 情感陪伴基础版：心情记录、低落/孤独/开心等场景话术、小鼋反馈。
- 适老化基础规范：大字号、强对比、少入口、语音播报预留。

### P1 增强

- “鼋气罐”积分：健康记录、亲情互动、社区活动获得积分，兑换模拟权益。
- 家庭共享相册：子女上传照片，老人端极简浏览。
- 远程协助：子女帮助老人设置提醒、联系人、隐私开关。
- 社区服务：活动报名、便民预约、工作人员处理状态。
- 情绪原因识别接口：先以规则和轻模型占位，后续替换为算法团队接口。
- 硬件真实接入：ESP32-S3 通过 MQTT 上报数据，后台统一解析。

### P2 试点期

- 多传感器融合：毫米波雷达 + 环境/柔性传感器联合判断。
- 语音助手：限定场景语音指令，如查询心率、联系儿子、呼叫帮助。
- 算法看板：误报率、召回率、设备在线率、响应时长。
- 数据授权与隐私日志：老人可查看谁访问了数据，可关闭部分监测项。
- 慢病管理扩展：血压、用药、复诊提醒。

## 5. 信息架构

### 老人端导航

- 首页：安全状态、SOS、小鼋提示、今日健康摘要。
- 健康：健康报告、健康日记、体征趋势。
- 联络：家庭聊天室、共享相册、视频/语音入口。
- 服务：社区活动、便民预约、银发课堂。
- 小鼋：心情记录、陪伴互动、鼋气罐。

老人端底部导航建议最多 4 个主入口，把“小鼋”作为中心入口或首页常驻按钮，避免入口过多。

### 子女端导航

- 守护：老人状态、告警、设备在线情况。
- 报告：日报、周报、趋势、异常解释。
- 关怀：相册、消息、提醒、陪伴建议。
- 设置：绑定关系、通知规则、隐私授权。

### 社区网页后台

- 告警中心：待处理、处理中、已关闭。
- 老人档案：风险等级、设备状态、联系人。
- 服务工单：预约、分派、处理记录。
- 数据统计：响应时长、服务完成率、设备在线率。

## 6. 核心业务流程

### 跌倒/异常告警

1. 设备端上报疑似事件。
2. 后端进行规则判断或调用算法接口二次确认。
3. 生成告警事件，写入日志。
4. 同步通知子女端和社区端。
5. 子女或社区工作人员确认响应。
6. 记录处理结果，关闭告警。
7. 纳入健康报告和试点指标统计。

### SOS 呼叫

1. 老人按硬件按钮或点击小程序 SOS。
2. 后端立即创建高优先级事件。
3. 并行触发微信通知、短信/电话、社区后台弹窗。
4. 若指定时间内无人确认，升级通知备用联系人。
5. 全流程记录时间戳和处理人。

### 情感陪伴

1. 老人记录心情或发起语音/文字互动。
2. 系统识别情绪类型和可能原因。
3. 小鼋给出安抚、鼓励、提醒或亲情联络建议。
4. 必要时向子女端生成“关怀建议”，但避免泄露过度隐私。
5. 互动结果进入心情趋势和鼋气罐积分。

## 7. 数据模型草案

关键实体：

- User：账号、手机号、角色、状态。
- ElderProfile：老人档案、紧急联系人、健康基础信息、隐私偏好。
- FamilyBinding：老人与子女绑定关系、权限级别。
- Device：设备编号、类型、在线状态、固件版本、绑定老人。
- SensorReading：心率、呼吸、活动状态、环境数据、上报时间。
- AlertEvent：告警类型、等级、来源、状态、处理记录。
- HealthReport：日报/周报汇总、趋势、建议。
- MoodRecord：心情、文本/语音、识别标签、回应内容。
- CompanionInteraction：小鼋互动记录、话术场景、满意度。
- YuanqiPointLedger：积分来源、积分变化、兑换记录。
- CommunityServiceOrder：服务类型、预约时间、处理人员、状态。

## 8. 里程碑

### 第 1 阶段：产品骨架与视觉规范，1-2 周

- 明确三类用户角色和 MVP 功能边界。
- 输出低保真原型、页面清单、适老化 UI 规范。
- 建立仓库结构、前后端基础工程、接口规范。

交付物：

- 页面流程图。
- 适老化设计规范。
- API 草案。
- 可运行空壳应用。

### 第 2 阶段：MVP 闭环，3-5 周

- 完成老人端、子女端核心页面。
- 完成设备模拟器、传感数据上报、告警闭环。
- 完成健康报告和基础小鼋互动。

交付物：

- 可演示的移动端应用。
- 可演示的网页/社区后台。
- 设备模拟脚本。
- 第一版测试用例。

### 第 3 阶段：硬件与算法接入，6-8 周

- 接入 ESP32-S3 + 毫米波雷达数据。
- 统一设备协议和事件解析。
- 对接跌倒检测、情绪识别、语音播报等算法接口。

交付物：

- 硬件接入说明。
- 算法接口说明。
- 端到端告警演示视频或脚本。

### 第 4 阶段：试点与迭代，9-12 周

- 在模拟或真实社区场景中测试。
- 统计误报、响应时长、任务完成率、满意度。
- 优化老人端交互、子女通知策略和社区后台。

交付物：

- 试点数据报告。
- 用户反馈清单。
- 迭代版本。
- 结题展示材料。

## 9. 验收指标

产品指标：

- 老人端核心任务 2 步内完成：SOS、查看安全状态、联系家人、记录心情。
- 老人端主要按钮热区不小于 44px，关键文字高对比显示。
- 子女端能在 10 秒内看到新告警并确认处理。
- 社区端告警列表状态清晰，可追溯处理记录。

技术指标：

- 设备模拟数据上报延迟小于 3 秒。
- 告警推送链路可重试、可追踪、可去重。
- 设备离线、弱网、重复上报、误触发都有处理逻辑。
- 真实硬件接入后，跌倒检测召回率、精确率、误报率按试点数据持续记录。

研究/大创指标：

- 形成产品原型、技术文档、用户测试报告。
- 支撑论文、专利或结题材料中的系统演示。
- 保留试点过程数据，能解释设计如何回应“增权 + 共情”理论。

## 10. 近期开发建议

建议下一步先做“可演示 MVP”，用模拟设备把业务闭环跑通，再逐步替换真实硬件和算法。第一版不要被高精度算法阻塞。

近期任务顺序：

1. 选定技术栈：建议 uni-app + Vue 3 + TypeScript + FastAPI + PostgreSQL。
2. 画页面原型：先画老人端 6 个核心页面、子女端 5 个核心页面、社区后台 3 个核心页面。
3. 定接口协议：设备上报、告警事件、健康报告、心情互动、积分流水。
4. 建工程：前端、后端、数据库迁移、设备模拟器。
5. 做闭环：模拟跌倒/SOS -> 后端生成告警 -> 子女/社区端接收 -> 确认处理 -> 报告归档。
6. 做适老化打磨：大字号、高对比、语音提示、减少层级。
7. 接真实硬件和算法接口。
